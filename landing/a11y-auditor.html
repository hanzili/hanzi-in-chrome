Real-browser accessibility audit

Tabs through your page, checks contrast on rendered elements, and verifies ARIA labels in context.

How it works

Paste a URL. Get a WCAG report.

1

Reviews your code first

Before opening the browser, it scans your codebase for ARIA usage, semantic HTML, alt text, form labels, and heading hierarchy.

2

Checks visual accessibility

Opens your page in a real browser to measure color contrast, font sizes, focus indicators, and touch target dimensions on rendered elements.

3

Tests keyboard navigation

Tabs through the entire page to verify tab order, checks for keyboard traps in modals, and tests that all interactive elements are keyboard-operable.

4

Verifies ARIA and semantics

Checks landmarks, form label associations, dynamic content announcements via aria-live, and that images and icons have accessible names.

5

Screenshots every issue

Each finding comes with a screenshot showing exactly what the page looks like, so you can see the problem without opening a browser yourself.

6

Delivers a prioritized report

Issues are categorized by severity (Critical, Serious, Moderate, Minor) with WCAG criteria, specific element locations, and code-level fix suggestions.

Why this is different

Not another Lighthouse scan.

Tests interactive state

Existing tools scan the DOM. This agent actually tabs through your page, tests focus indicators, and catches keyboard traps in modals and dropdowns.

Handles dynamic content

SPAs, async-loaded content, modals, and client-side routing all get tested in their rendered state, not just the initial HTML payload.

Actionable code fixes

Each issue includes the file path, WCAG criterion, and a specific code change. Not just "fix your contrast" but exactly which element and what value to use.
